The part nobody else has

The dashed column above tells the story. Here is what that one green cell actually gives you.

Built-in MCP server

Point Claude, Cursor or Copilot straight at your scheduler.

cronstable speaks the Model Context Protocol natively — served at POST /mcp over Streamable HTTP and through a cronstable mcp stdio bridge for desktop clients. It's hand-rolled in pure Python against MCP revision 2025-11-25, so enabling it adds zero new dependencies. An agent gets look-but-don't-touch access until you deliberately hand it the keys.

# wire it to Claude Code — remote or local
claude mcp add --transport http \
  cronstable https://host/mcp \
  --header "Authorization: Bearer $TOKEN"

$ cronstable mcp --check
mcp check: ok — protocol 2025-11-25,
  23 tool(s) at http://127.0.0.1:8080/mcp

Only cronstable ships a native MCP server

Across every tool surveyed, it's the sole native agent interface. Airflow and Kubernetes reach it only through third-party, whole-system MCP add-ons — Airflow's own is still a draft proposal.

Methodology. Rows are cronstable's own capability set. Competitor cells were researched against each project's official documentation, README and changelog, then adversarially re-verified — every "not available" mark survived a dedicated attempt to refute it. Four claims were revised on evidence: dkron per-job resource gauges and Cronicle crontab-import / Time-Machine backfill were softened to "partial", and Airflow's statsd/OpenTelemetry metrics were confirmed native. Airflow's own MCP server is a draft proposal (AIP-91) with only third-party REST wrappers today, so it is graded "add-on" like Kubernetes; "classic crontab" is scored "no" for tools that accept a cron expression but do not ingest crontab files.

Compared: yacron, supercronic, Ofelia, dkron (OSS + Pro noted), Cronicle, native Kubernetes CronJob, and Apache Airflow (the heavyweight orchestrator, shown for scale).
